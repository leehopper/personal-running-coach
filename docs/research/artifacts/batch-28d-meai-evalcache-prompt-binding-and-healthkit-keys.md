# Part 1 — Eval-cache ↔ prompt-template binding (M.E.AI.Evaluation 10.6.0)

# Cache-Key Derivation in M.E.AI.Evaluation 10.6.0 and the Lowest-Overhead Prompt-Change Detector for RunCoach

## TL;DR
- In Microsoft.Extensions.AI 10.6.0 the response cache key is a hash of the JSON serialization of **the full ChatMessage[] AND the entire ChatOptions object AND an `additionalValues` span** — and `ChatOptions.AdditionalProperties` is NOT marked `[JsonIgnore]`, so it **does** participate in the key. The batch-17c recommendation (`options.AdditionalProperties["runcoach.prompt_version"]`) therefore **works as written**, but 10.6.0 also ships a cleaner first-class knob, `DistributedCachingChatClient.CacheKeyAdditionalValues`, that you should prefer.
- The framing is **correct**: the system prompt text already travels inside `messages[]` (and `ChatOptions.Instructions`, also serialized), so any prompt edit already changes the key → a clean MISS in Replay mode → CI failure. Correctness is not at risk; the only real gap is *timing* — the developer learns about it as a post-merge CI Replay failure rather than locally at commit time.
- Recommended detector: a **committed SHA-256 sentinel manifest of `Prompts/*.yaml`, checked by a glob-scoped lefthook `pre-commit` hook and regenerated by `rerecord-eval-cache.sh`**, with a tiny pre-test fixture assertion in `EvalTestBase` as a CI / `--no-verify` backstop.

## Key Findings

### 1. Verified cache-key inputs (Microsoft.Extensions.AI 10.6.0)
The authoritative reference XML (assembly version 10.6.0.0) defines:

```
protected override string GetCacheKey(
    IEnumerable<ChatMessage> messages,
    ChatOptions? options,
    scoped ReadOnlySpan<object?> additionalValues);
```

with the verbatim remark: *"The messages, options, and additionalValues are serialized to JSON using JsonSerializerOptions in order to compute the key."* and *"The generated cache key is not guaranteed to be stable across releases of the library."*

So the key is `hash(JSON(messages) + JSON(options) + JSON(additionalValues))`:
- **(a) ChatMessage[] — YES.** The full messages array (including any system-prompt message text) is serialized into the key.
- **(b) ChatOptions — YES.** The whole `ChatOptions` object is serialized.
- **(c) ChatOptions.AdditionalProperties — YES, it participates.** This is the decisive finding, verified directly against the 10.6.0.0 `ChatOptions` reference XML: properties that must NOT affect serialization carry `[System.Text.Json.Serialization.JsonIgnore]` — specifically `Tools`, `RawRepresentationFactory`, and the experimental `ContinuationToken` / `AllowBackgroundResponses`. `AdditionalProperties` (type `AdditionalPropertiesDictionary`) carries **no `[JsonIgnore]` attribute**, so it is serialized as part of `JSON(options)` and is therefore part of the cache key.

### 2. batch-17c verification
Injecting `options.AdditionalProperties["runcoach.prompt_version"] = "..."` **does change the cache key** in 10.6.0, because `AdditionalProperties` is serialized into `JSON(options)`. The serialization-fidelity caveat in the docs — that *"Object values in AdditionalProperties will deserialize as JsonElement rather than as the original type"* — is (i) attached in the class remarks to `ChatMessage.AdditionalProperties` / `ChatMessage.RawRepresentation`, and (ii) irrelevant to keying regardless, because only the *write* (serialize) side feeds the key, not a round-trip. So batch-17c is technically valid.

However, 10.6.0 ships a purpose-built knob that is cleaner than overloading `AdditionalProperties`: `DistributedCachingChatClient.CacheKeyAdditionalValues` (`public IReadOnlyList<object>? CacheKeyAdditionalValues { get; set; }`), documented verbatim as *"Gets or sets additional values used to inform the cache key employed for storing state."* with the remark *"Any values set in this list will augment the other values used to inform the cache key."* This maps directly to the `additionalValues` span in `GetCacheKey`. **Prefer `CacheKeyAdditionalValues` over `AdditionalProperties`** when you have a handle on the caching client — it's the supported, self-documenting way to add custom values to the key.

### 3. The real problem is timing, not correctness
Because prompt text is already part of `messages[]`, editing a prompt already busts the key. In Replay mode (`EVAL_CACHE_MODE=Replay`, zero live calls), a busted key = no matching committed entry = MISS = test/CI failure. That is *correct* behavior — the framing in the task is right. The gap is purely **when** the developer finds out: today it surfaces as a confusing Replay failure on the next push, after the change merged. A local detector moves that signal to commit time with an actionable message.

### 4. Recommended detector and placement
Recommend **(a) the prompt-hash sentinel manifest**, placed primarily in a **lefthook `pre-commit` hook** and regenerated by `rerecord-eval-cache.sh`, with a **pre-test fixture assertion in `EvalTestBase`** as the CI / `--no-verify` backstop.

## Details

**Why (a) beats (b) and (c):**
- **(b) Version-into-key + runtime check** is the weakest. Busting the key does not *detect* anything earlier than a plain prompt edit already does (both produce a MISS). The "runtime check that active version matches recorded version" only fires when tests run — no earlier than a fixture — and it depends on a human remembering to bump the `version:` field. Forgetting to bump = silent again, which is the exact failure mode you're trying to eliminate. It adds a moving part without closing the gap.
- **(c) Pre-commit hook that re-runs affected evals in Replay** does close the gap but is the heaviest option: it compiles and runs `dotnet test` on every commit touching a prompt — seconds to tens of seconds — duplicating what `prepush-eval-check.sh` already does at push time.
- **(a) Sentinel manifest** is near-instant (hashing a handful of small YAML files), requires no test build, is deterministic, and fails locally at commit with a clear remediation message. Crucially, it encodes the exact invariant you care about: *"the committed cache was last recorded against these prompt contents."* The manifest is written by the same script that re-records the cache and patches the TTL, so the cache and the manifest move together atomically; any prompt edit not accompanied by a re-record is flagged immediately.

**Placement justification:** A `pre-commit` hook catches the issue earliest and, glob-scoped to `Prompts/*.yaml`, costs nothing on the vast majority of commits (lefthook skips the command entirely when no staged file matches the glob). Its two weaknesses — bypassable with `git commit --no-verify`, and git hooks don't run in CI — are exactly covered by a tiny fixture assertion in `EvalTestBase`, which runs whenever the eval suite runs (locally and in CI) and throws a clear message *before* the opaque Replay-miss error. The two share the same ~10 lines of hashing logic. This fits a solo-builder repo that already has lefthook hooks (`prepush-eval-check.sh`, `rerecord-eval-cache.sh`).

**Sketch — `check-prompt-hashes.sh` (pre-commit, <25 LOC bash):**
```bash
#!/usr/bin/env bash
set -euo pipefail
MANIFEST="Prompts/.prompt-hashes.sha256"
TMP="$(mktemp)"
# Bare filenames so the manifest is location-stable; sorted for determinism.
( cd Prompts && sha256sum *.yaml ) | sort > "$TMP"   # macOS: shasum -a 256
if ! diff -q "$MANIFEST" "$TMP" >/dev/null 2>&1; then
  echo "❌ Prompts/*.yaml changed but the committed eval cache was not re-recorded."
  echo "   The committed disk cache will MISS in Replay mode and CI will fail."
  echo "   Fix: run ./rerecord-eval-cache.sh (re-records cache + refreshes manifest), then re-stage."
  diff "$MANIFEST" "$TMP" || true
  rm -f "$TMP"; exit 1
fi
rm -f "$TMP"
```

**Wire-up in `lefthook.yml`:**
```yaml
pre-commit:
  commands:
    prompt-cache-sync:
      glob: "Prompts/*.yaml"
      run: ./check-prompt-hashes.sh
```

**Append to `rerecord-eval-cache.sh` (after cache record + TTL patch to 9999-12-31):**
```bash
( cd Prompts && sha256sum *.yaml ) | sort > Prompts/.prompt-hashes.sha256
git add Prompts/.prompt-hashes.sha256
```

**Backstop — `EvalTestBase` fixture assertion (<25 LOC C#):**
```csharp
static EvalTestBase()
{
    var dir = Path.Combine(RepoRoot(), "Prompts");
    var manifest = Path.Combine(dir, ".prompt-hashes.sha256");
    if (!File.Exists(manifest)) return;            // manifest optional
    var sb = new StringBuilder();
    foreach (var f in Directory.GetFiles(dir, "*.yaml")
                               .OrderBy(p => Path.GetFileName(p), StringComparer.Ordinal))
    {
        var hex = Convert.ToHexString(SHA256.HashData(File.ReadAllBytes(f))).ToLowerInvariant();
        sb.Append(hex).Append("  ").Append(Path.GetFileName(f)).Append('\n');
    }
    if (sb.ToString() != File.ReadAllText(manifest).Replace("\r\n", "\n"))
        throw new InvalidOperationException(
            "Prompts/*.yaml changed but the committed eval-cache manifest is stale. " +
            "Run ./rerecord-eval-cache.sh or the Replay cache will MISS.");
}
```
(Match the exact byte layout `sha256sum` produces — lowercase hex, two spaces, bare filename, `\n` — or normalize both sides before comparing.)

## Recommendations
1. **Adopt the sentinel manifest now.** Generate `Prompts/.prompt-hashes.sha256` once via `( cd Prompts && sha256sum *.yaml ) | sort`, commit it, and add `check-prompt-hashes.sh` + the `lefthook.yml` `pre-commit` entry above.
2. **Make `rerecord-eval-cache.sh` the single source of truth** by having it rewrite and `git add` the manifest immediately after it patches the TTL to 9999-12-31. This guarantees cache + manifest are always committed together, eliminating the silent-drift window.
3. **Add the `EvalTestBase` static-ctor backstop** so the check also runs in CI and survives `--no-verify`. Keep the remediation string identical to the hook's.
4. **Do not adopt option (b) as the detector.** If you nonetheless want the prompt version reflected in the key for forensic/debugging clarity, set `DistributedCachingChatClient.CacheKeyAdditionalValues = ["runcoach-prompts:" + manifestHash]` (preferred over `AdditionalProperties`); but note this is redundant with detection and forces a full cache re-record on every bump.
5. **Benchmarks/thresholds that would change the recommendation:** hashing cost stays negligible even at hundreds of prompts, so (a) remains cheapest. If you move to a team and need a non-bypassable gate, promote the `EvalTestBase` fixture assertion to the authoritative check and treat the pre-commit hook as convenience-only. If you start caring about *which* prompt changed (selective re-record), graduate to per-prompt manifest entries — already supported by the sketch's per-file lines.

## Caveats
- **Library-version coupling.** `DistributedCachingChatClient` lives in core Microsoft.Extensions.AI (10.6.0), which versions independently of Microsoft.Extensions.AI.Evaluation(.Reporting). `CacheKeyAdditionalValues` is present in 10.6.0.0 but may be absent in older 9.x previews. The documented warning that *"the generated cache key is not guaranteed to be stable across releases"* means **bumping any M.E.AI.* package can silently invalidate the entire committed cache**, exactly like a prompt edit — the sentinel manifest does NOT cover this, so pair package bumps with a `rerecord-eval-cache.sh` run.
- **Source-body verification.** The cache-key inputs, the `CacheKeyAdditionalValues` semantics, and the `ChatOptions.AdditionalProperties`-is-not-`[JsonIgnore]` fact are confirmed from the 10.6.0.0 reference XML and Microsoft Learn (both generated from source). The exact hashing primitive (SHA-256 vs. other) and the precise line in `CachingChatClient.GetResponseAsync` that folds `CacheKeyAdditionalValues` into the `additionalValues` span could not be retrieved as raw `.cs` source in this pass; neither affects any conclusion here. The Evaluation.Reporting layer drives caching through a `ResponseCachingChatClient` backed by `DiskBasedResponseCache` (an `IDistributedCache`), so the core JSON-serialize-and-hash keying applies; the framework's default cache TTL is 14 days (the behavior you patch to 9999-12-31).
- **Replay-miss behavior.** Whether a MISS in Replay mode throws or merely fails the assertion depends on how RunCoach configures the offline client; either way CI fails, which is the desired correctness property.
- **`ConversationId` disables caching.** Per Microsoft Learn, the default `EnableCaching` *"returns true as long as the options does not have a ConversationId set."* Ensure eval requests leave `ConversationId` null, or caching is bypassed entirely and Replay has nothing to serve.
- **`sha256sum` portability.** Linux CI provides `sha256sum`; macOS dev machines use `shasum -a 256`. Pick one and normalize output, or the manifest will appear "changed" across platforms and produce false failures.

# Part 2 — Canonical metric-key reconciliation (HealthKit / Strava / Garmin)

# Canonical Metric-Key Reconciliation for `WorkoutLog.Metrics` (JSONB)

A single canonical camelCase key set can and should be fixed now: with the unit decisions below, manual entry, Apple HealthKit, Strava, and Garmin can all write the SAME keys into the `Metrics` bag with no future migration. This document fixes key **names** and **units** plus the per-lap split object shape only — ingestion implementation, EF/JSONB storage mechanics, eval rubrics, and TTL/cache are out of scope.

> Note on internal docs: a Google Drive search for `DEC-033`, `batch-3c`, and `WorkoutMetricKeys` returned nothing accessible, so the project constraints (canonical meters/seconds; HealthKit as the post-MVP-0 priority; Garmin "Activity Summary" names as the given to reconcile against) are taken as stated in the task. Garmin names below are flagged by confidence because Garmin runs two different APIs whose field names differ (see Section 3).

## TL;DR
- **One canonical camelCase key set is viable now.** The only hard reconciliation costs are three unit gotchas: pace/speed, cadence, and stride length — all resolved below in a way consistent with the meters/seconds storage rule.
- **Store raw `distanceMeters` + `durationSeconds` always and derive pace as `paceSecPerKm`** (seconds-per-kilometer); never trust a source's own pace field. Store **cadence as full steps/min** (double Strava's single-leg API value). Store **stride length in meters** (convert Garmin, which reports centimeters in Connect / millimeters at the ANT+ level).
- **Five keys are intrinsically manual-only with no clean source equivalent:** `rpe`, `recoveryScore`, `sleepScore`, `terrain`, `weather`. This is expected, not a defect — they capture subjective/coaching context that wearables don't standardize, though Garmin can partially fill `sleepScore` and `recoveryScore` and HealthKit exposes partial `weather` metadata.

## Section 1 — HealthKit identifier table

HealthKit data falls into four shapes that determine how a single value lands in the `Metrics` bag: **sample-level** time series (`HKQuantitySample`), **workout-level aggregates** derived via `HKStatistics` / `HKWorkoutStatistics`, **category samples** (sleep), and **workout metadata keys** (`HKMetadataKey…`).

| Metric | HealthKit identifier | Native unit | Workout-level vs sample | Notes |
|---|---|---|---|---|
| Heart rate (instantaneous) | `HKQuantityTypeIdentifier.heartRate` | count/min (bpm) | sample-level | Average and max are **not** separate identifiers. |
| Average HR | **derived**: `HKStatistics.averageQuantity()` over `heartRate` | count/min | workout-level (statistic) | Use `HKWorkoutStatistics` or `HKStatisticsQuery` with `.discreteAverage`. Unit = `HKUnit.count().unitDivided(by: .minute())`. |
| Max HR | **derived**: `HKStatistics.maximumQuantity()` over `heartRate` | count/min | workout-level (statistic) | `.discreteMax` option. |
| Active energy (calories) | `HKQuantityTypeIdentifier.activeEnergyBurned` | kcal | sample-level; summed | Distinct from `basalEnergyBurned` (resting energy). Use **active** for "calories." |
| Distance | `HKQuantityTypeIdentifier.distanceWalkingRunning` | meters | sample-level; summed | Native unit is meters. |
| Running speed | `HKQuantityTypeIdentifier.runningSpeed` | m/s | sample-level | iOS 16+ (2022). |
| Running power | `HKQuantityTypeIdentifier.runningPower` | watts | sample-level | iOS 16+. ("the rate of work required for the runner to maintain their speed.") |
| Running stride length | `HKQuantityTypeIdentifier.runningStrideLength` | meters | sample-level | iOS 16+. ("the distance covered by a single step while running.") |
| Vertical oscillation | `HKQuantityTypeIdentifier.runningVerticalOscillation` | centimeters | sample-level | iOS 16+. ("pelvis vertical range of motion during a single running stride.") |
| Ground contact time | `HKQuantityTypeIdentifier.runningGroundContactTime` | milliseconds | sample-level | iOS 16+. |
| Cadence | **no dedicated identifier** | — | — | Confirmed against Apple's `HKQuantityTypeIdentifier` list, which includes `runningSpeed`, `runningPower`, `runningStrideLength`, `runningVerticalOscillation`, `runningGroundContactTime` but **no running-cadence identifier**. Derive from speed ÷ stride length, or steps ÷ time. |
| HRV | `HKQuantityTypeIdentifier.heartRateVariabilitySDNN` | ms | sample-level | **SDNN is the only HRV type HealthKit exposes** ("the standard deviation of heartbeat intervals," introduced iOS 11.0/2017). Apple Watch uses SDNN, not RMSSD, as its native HRV metric. |
| Resting HR | `HKQuantityTypeIdentifier.restingHeartRate` | count/min | sample-level (daily) | Daily metric, not a per-workout aggregate. |
| VO₂max | `HKQuantityTypeIdentifier.vo2Max` | mL/(kg·min) | sample-level (periodic) | "the maximal oxygen consumption during exercise." |
| Flights climbed | `HKQuantityTypeIdentifier.flightsClimbed` | count | sample-level; summed | **Not** the same as elevation gain. |
| Elevation ascended | `HKMetadataKeyElevationAscended` | length unit (meters) | **workout metadata** | "the cumulative elevation ascended during a workout." Set on a workout / segment / distance sample — **not** a quantity-type time series. |
| Sleep | `HKCategoryTypeIdentifier.sleepAnalysis` | category (stage durations) | category samples | Stages: `inBed`, `asleepUnspecified`, `asleepCore`, `asleepDeep`, `asleepREM`, `awake`. **No unified numeric sleep score** — only per-stage durations you sum yourself. |
| Weather temperature | `HKMetadataKeyWeatherTemperature` | temperature quantity | workout metadata | iOS 10+. "the overall temperature during the workout." |
| Weather humidity | `HKMetadataKeyHumidity` | percent | workout metadata | iOS 10+. |
| Weather condition | `HKMetadataKeyWeatherCondition` | enum | workout metadata | iOS 10+. |
| RPE | **none** | — | — | App-supplied only. |
| Recovery / readiness score | **none** | — | — | Apple exposes no recovery/readiness score (contrast WHOOP and Garmin Body Battery). |

Critical HealthKit facts:
- **Average and max HR are derived statistics, not stored identifiers.** `HKWorkoutStatistics`/`HKStatistics` expose `averageQuantity()`, `maximumQuantity()`, `minimumQuantity()`, `sumQuantity()`. This means a single `hrAvg`/`hrMax` value is computed at import, not read from a named field.
- **Sleep is not a score in HealthKit** — it is a category time series of stages; any "score" must be computed or comes from another source.
- **Elevation ascent is workout metadata** (`HKMetadataKeyElevationAscended`) expressed in meters, not a sample stream.

## Section 3 — THE DELIVERABLE: final canonical key set

**Pace/speed decision (explicit):** Always store raw `distanceMeters` and `durationSeconds`; store the derived runner-facing pace as **`paceSecPerKm` (seconds per kilometer)**. Reasons: (1) it is trivially derived from canonical meters/seconds (`paceSecPerKm = durationSeconds / (distanceMeters/1000)`); (2) seconds-per-kilometer keeps values in a human-meaningful integer range (e.g., 300 = 5:00/km) and avoids the tiny-float problem of seconds-per-meter; (3) it is recomputed identically regardless of whether the source gave m/s (Strava/HealthKit) or min/km (Garmin), so no source's pace field is ever trusted directly.

| Canonical key | Canonical unit | HealthKit source | Garmin name | Strava field | Workout/sample | Disagreement flag |
|---|---|---|---|---|---|---|
| `rpe` | integer 1–10 | none | none | `perceived_exertion` (1–10) | workout | App-supplied; Strava's scale aligns 1:1. |
| `hrAvg` | bpm | derived statistic over `heartRate` | `averageHeartRateInBeatsPerMinute` | `average_heartrate` | workout (derived) | None — all bpm. |
| `hrMax` | bpm | derived statistic over `heartRate` | `maxHeartRateInBeatsPerMinute` | `max_heartrate` | workout (derived) | None. |
| `calories` | kcal | `activeEnergyBurned` (summed) | `activeKilocalories` | `calories` | workout | **None — all three are kcal. ✔ confirmed.** |
| `splits` | array of objects | derived from `HKWorkoutEvent .lap`/`.segment` | lap data | `laps` / `splits_metric` | workout | ⚠ Shape differs by source — see §4. |
| `hrv` | ms (SDNN) | `heartRateVariabilitySDNN` | `LastNightAvg` (overnight, ms) | none | sample / daily | ⚠ Not a per-run aggregate; daily/contextual. Garmin gives overnight only; SDNN vs Garmin RMSSD basis differ. |
| `sleepScore` | integer 0–100 | none (only stages) | `OverallSleepScoreValue` (0–100) + `OverallSleepScoreQualifierKey` | none | daily | ⚠ HealthKit has no score; Garmin partner Sleep Summary does. |
| `recoveryScore` | integer 0–100 | none | Body Battery (0–100, in Stress Detail Summary) | none | daily | ⚠ App/Garmin-only. Body Battery is "Garmin's proprietary energy level indicator, scored from 0 to 100… combines HRV analysis, stress detection and sleep quality." |
| `weather` | object | `HKMetadataKeyWeatherTemperature` / `…Humidity` / `…WeatherCondition` | not in activity summary | none (UI only) | workout metadata | ⚠ Only partial sources; mostly manual. |
| `terrain` | string enum | none | none | none | workout | Manual-only everywhere. |
| `cadence` | steps/min (full, both feet) | none (derive: speed ÷ stride length) | `averageRunCadenceInStepsPerMinute` | `average_cadence` × 2 | workout | ⚠ Strava single-leg doubling; HealthKit has no cadence identifier. |
| `cadenceMax` (optional) | steps/min | none | `maxRunCadenceInStepsPerMinute` | n/a | workout | Optional reserve. |
| `elevationGain` | meters | `HKMetadataKeyElevationAscended` | `totalElevationGainInMeters` | `total_elevation_gain` | workout | **None — all meters. ✔** |
| `power` | watts | `runningPower` (sample → avg) | sample `PowerInWatts`; internal-API `avgPower` (summary) | `average_watts` | sample → workout avg | ⚠ Garmin's **official** partner API exposes power only as per-second `PowerInWatts` samples, not a summary average; summary `avgPower` exists only in Garmin's **internal** Connect web API. |

**Rationale where sources disagree:**
- **cadence → full steps/min.** Matches the HealthKit-derived value and Garmin's `averageRunCadenceInStepsPerMinute`; document that Strava's API `average_cadence` must be doubled (a working steps approximation is `average_cadence × 2 × moving_time/60`).
- **strideLength → meters.** HealthKit's `runningStrideLength` is native meters and consistent with the meters storage rule. **Flag:** Garmin reports stride length in **centimeters** in Connect (e.g., ~106 for a typical runner) and in **millimeters** at the ANT+ Running Dynamics stream level (Connect IQ `AntPlus.RunningDynamicsData`: "Step Length (0 – 8191 mm)"). Convert Garmin to meters on import (÷100 from Connect, ÷1000 from ANT+).
- **pace → `paceSecPerKm` derived.** Never store a source's pace field directly; recompute from canonical distance/duration.
- **energy → kcal, no conversion.** All three sources use kcal (confirmed).
- **hrv → ms SDNN.** Flag that HealthKit's SDNN and Garmin's RMSSD-based overnight value are different HRV statistics, and that neither is a per-run number — treat `hrv` as daily context attached to a workout.

**Running-dynamics keys — recommendation: DEFINE THE NAMES NOW, POPULATE LATER.** Because constants can never change once history is written and wearable integrations make these cheap, reserve the names at creation even though MVP manual entry won't use them. Reserved keys and canonical units:
- `verticalOscillation` — **centimeters** (HealthKit `runningVerticalOscillation` = cm; Garmin internal `avgVerticalOscillation` = cm — **agree**; ANT+ raw is mm, convert).
- `groundContactTime` — **milliseconds** (HealthKit `runningGroundContactTime` = ms; Garmin internal `avgGroundContactTime` = ms — **agree**).
- `strideLength` — **meters** (HealthKit native m; Garmin cm/mm — ⚠ convert on import).

This costs nothing now (the constants are just defined) and guarantees no migration when HealthKit running-form ingestion ships.

## Section 4 — Canonical splits / lap object shape

**Recommended minimal-but-future-proof per-lap object** (array of objects under `splits`):

```jsonc
"splits": [
  {
    "index": 1,               // int, 1-based
    "distanceMeters": 1000,   // meters
    "durationSeconds": 240,   // seconds — use elapsed_time; document the choice
    "paceSecPerKm": 240,      // derived, seconds per kilometer
    "averageHeartRate": 165   // optional, bpm
  }
]
```

Field-by-field source mapping:

| Canonical lap field | HealthKit derivation | Strava `laps[]` field | Strava `splits_metric[]` field |
|---|---|---|---|
| `index` | order of `HKWorkoutEvent` type `.lap` | `lap_index` | `split` |
| `distanceMeters` | derived (sum distance between lap start/end dates) | `distance` (m) | `distance` (m) |
| `durationSeconds` | derived (end − start dates) | `elapsed_time` / `moving_time` (s) | `elapsed_time` / `moving_time` (s) |
| `paceSecPerKm` | derived | from `average_speed` (m/s → `1000/speed`) | from `average_speed` |
| `averageHeartRate` (optional) | derived statistic over the interval | `average_heartrate` | `average_heartrate` |

**HealthKit reconciliation note:** laps are `HKWorkoutEvent` of type `.lap`/`.segment` and carry only start/end dates (plus optional metadata such as swim stroke style). They are **not richly typed objects**, so distance, duration, and pace must be **derived** from the event date interval and the underlying samples. Multi-sport workouts model sub-activities/intervals as `HKWorkoutActivity` (queried via `statistics(for:)`). The array-of-objects design therefore matches what a Strava import populates directly and what a HealthKit import can populate after derivation — confirming the design is correct for later ingestion.

**Optional future lap fields to reserve if desired** (define names now, populate later): `maxHeartRate` (bpm; Strava lap `max_heartrate`), `elevationDifferenceMeters` (Strava split `elevation_difference` / lap `total_elevation_gain`, meters), `averagePower` (watts), `movingTimeSeconds` (if you decide to keep both moving and elapsed). Keep the **required** set to the five fields above to stay minimal.

## Section 5 — Manual-only canonical keys (expected never to auto-populate)

These keys have no clean cross-source equivalent and will normally be hand-entered; this is by design, not a gap:
- **`rpe`** — subjective; no HealthKit equivalent. Strava's `perceived_exertion` (1–10) aligns and could backfill it if/when Strava is integrated, but otherwise manual.
- **`recoveryScore`** — no HealthKit or Strava equivalent. Only Garmin Body Battery (0–100) could ever populate it.
- **`sleepScore`** — HealthKit yields only stage durations (no score); Garmin's partner Sleep Summary `OverallSleepScoreValue` (0–100) could populate it; Strava has none.
- **`terrain`** — no source exposes a terrain field; always manual (string enum).
- **`weather`** — only partial: HealthKit workout metadata (`HKMetadataKeyWeatherTemperature` / `…Humidity` / `…WeatherCondition`) can enrich it; Strava shows weather only in the UI. Treat as manual with optional enrichment.

## Recommendations (staged, with thresholds)

1. **Lock the key set and units now** (this document): define `WorkoutMetricKeys` with `rpe`, `hrAvg`, `hrMax`, `calories`, `splits`, `hrv`, `sleepScore`, `recoveryScore`, `weather`, `terrain`, `cadence`, `elevationGain`, `power`, plus reserved-but-unused `verticalOscillation`, `groundContactTime`, `strideLength` (and optional `cadenceMax`). Encode units in the constant names' documentation, not the keys (e.g., `paceSecPerKm` is the rare case where the unit is in the name because it is a derived field with no canonical raw source).
2. **Write the unit contract into the constants file as XML doc comments** for the three gotcha keys: cadence = full steps/min (double Strava API); strideLength = meters (convert Garmin cm/mm); pace = always derived `paceSecPerKm`, never trust source pace.
3. **For the HealthKit MVP-0+ integration**, build the import to (a) compute `hrAvg`/`hrMax` via `HKStatistics`, (b) sum `activeEnergyBurned` for `calories` and `distanceWalkingRunning` for distance, (c) read `HKMetadataKeyElevationAscended` for `elevationGain`, and (d) derive `splits` from `.lap` events. **Threshold to revisit:** if HealthKit ever ships a `runningCadence` quantity identifier, prefer it over the derived cadence (but keep the same canonical key/unit — no migration).
4. **Defer running-dynamics population** until there is a confirmed wearable source feeding them; the reserved names mean this is a zero-migration switch-on.
5. **Treat `hrv`, `sleepScore`, `recoveryScore` as daily/context fields**, not per-run aggregates; if you later attach them to a workout, document that they are point-in-time context (Garmin overnight values), not measured during the run.
6. **Flag for Garmin specifically:** when Garmin integration is scoped, confirm whether you are consuming the **official partner Health/Activity API** (verbose `…InMeters` names; no summary running-dynamics or summary power) or the **internal Connect web API** (`avgVerticalOscillation`, `avgGroundContactTime`, `avgStrideLength`, `avgPower`). The summary running-dynamics and average-power fields exist **only** in the internal API; the official partner API exposes power only as per-second `PowerInWatts` samples.

## Caveats
- **Internal decision docs (`DEC-033`, `batch-3c`) were not retrievable** from Drive; the project constraints are taken as stated. If those docs specify a different pace convention or already-chosen names, reconcile against them before locking constants.
- **Garmin field names carry mixed confidence.** High confidence: `averageHeartRateInBeatsPerMinute`, `maxHeartRateInBeatsPerMinute`, `averageRunCadenceInStepsPerMinute`, `maxRunCadenceInStepsPerMinute`, `averageSpeedInMetersPerSecond`, `maxSpeedInMetersPerSecond`, `distanceInMeters`, `durationInSeconds`, `activeKilocalories`, `totalElevationGainInMeters`, `totalElevationLossInMeters`, `averagePaceInMinutesPerKilometer` (partner Activity API); `avgVerticalOscillation` (cm), `avgGroundContactTime` (ms), `avgStrideLength` (cm), `OverallSleepScoreValue` (internal/Sleep Summary). **Lower/medium confidence — verify before relying:** Garmin summary running-power field name (`avgPower`/`maxPower` are the internal-API community convention but were not verbatim-confirmed for a running activity); `averagePowerInWatts` is **not** a real Garmin field and must not be used. Body Battery per-day peak/low field names could not be confirmed (the documented location is the Stress Detail Summary time-series).
- **Stride-length unit ambiguity:** Garmin Connect UI/internal API uses centimeters while the ANT+ Running Dynamics stream uses millimeters; pin the exact import unit when wiring the source, then convert to canonical meters.
- **HealthKit running-form types require iOS 16+ (2022)** and a compatible Apple Watch; older workouts will lack them, so these keys will be sparsely populated even after integration.
- **`hrv` semantics differ across sources** (HealthKit SDNN vs Garmin RMSSD-based overnight); do not treat a stored `hrv` value as directly comparable across data origins without recording the source.
