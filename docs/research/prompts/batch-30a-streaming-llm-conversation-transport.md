# R-082: Streaming LLM coach responses — transport, RTK Query integration, and event-sourced persistence reconciliation

## Context

Slice 4 (Open Conversation) of RunCoach turns the read-only Slice 3 chat panel into an interactive, streaming coaching surface: the user types a question and sees the assistant's response render token-by-token. The conversation **core** (streaming endpoint + interactive panel + turn persistence) is blocked on this research; the parallel voice/persona re-tune and onboarding/prompt fixes are not.

Concrete stack facts established by codebase exploration (do not re-derive — design against these):

- **Backend:** ASP.NET Core controllers on **.NET 10 / C# 14**. Auth is the `CookieOrBearer` policy — `__Host-`-prefixed cookie session + antiforgery for the web SPA today, JWT bearer reserved for a future iOS client. Marten event-sourcing on PostgreSQL JSONB.
- **LLM adapter:** All calls route through `ICoachingLlm` (today **request/response only** — `GenerateAsync` / `GenerateStructuredAsync<T>`; **no streaming method**). The Microsoft.Extensions.AI layer is already present and a `SanitizationAuditChatClient : DelegatingChatClient` already implements `GetStreamingResponseAsync(...) : IAsyncEnumerable<ChatResponseUpdate>` (forwarding to the inner client / Anthropic streaming API) but it is **not yet exposed through `ICoachingLlm`**. The Anthropic SDK supports streaming natively.
- **Persistence:** Conversation turns are **event-sourced** — appended to a Marten stream and projected (inline) into a read-model. Today that read-model (`ConversationLogView`) is keyed by **`PlanId`** (per-plan, not per-user) and holds only deterministic adaptation/safety turns. Open-conversation user+assistant turns are net-new. (Whether the projection stays plan-scoped or becomes user-scoped is an **adjacent** decision — see the persistence note in Deliverables — but the streaming pattern must not foreclose it.)
- **Frontend:** React 19 + TypeScript (strict), Redux Toolkit + **RTK Query**, Vite SPA. The existing `conversation.api.ts` is a normal RTK Query endpoint (`getConversationTurns`) with a `Conversation` cache tag invalidated on workout-log create. **RTK Query is not stream-native** — the Slice 4 requirements doc flags this as the single biggest implementation risk.

## Research Question

**Primary:** What is the right end-to-end pattern to stream an LLM coach response from ASP.NET Core (.NET 10) to a React 19 + RTK Query SPA, given event-sourced (Marten) persistence and `CookieOrBearer` auth — covering wire transport, client integration, server pipeline, persistence reconciliation, and mid-stream safety/error handling as one coherent design?

Sub-questions that make the answer actionable:

1. **Transport.** For a single, unidirectional, server→client token stream per turn on this exact stack, compare **SSE vs. chunked HTTP / NDJSON vs. WebSocket vs. SignalR**. Weigh: compatibility with `CookieOrBearer` (note `EventSource` cannot set an `Authorization` header — does the cookie path sidestep this, and what does the future bearer/iOS path need?), antiforgery interplay, reverse-proxy / Kestrel response buffering, reconnection semantics, cancellation, browser support in 2026, and implementation complexity for a solo dev. **Recommend one** and justify.
2. **RTK Query integration.** Evaluate (a) RTK Query's documented `onCacheEntryAdded` streaming-updates pattern, (b) **raw `fetch` + `ReadableStream` reader + local React state** for the in-flight streaming turn only (keeping the rest of the app on RTK Query), and (c) a streaming-aware custom `baseQuery`. Which is the maintained 2026 idiom for token-by-token chat rendering? Critically: **how does the completed turn reconcile with the `getConversationTurns` RTK Query cache** (invalidate-and-refetch vs. optimistic merge) so the panel shows a stable, server-authoritative history once the stream ends — without a flash, a duplicate, or a lost turn?
3. **Server pipeline.** How to wire the existing `IAsyncEnumerable<ChatResponseUpdate>` (M.E.AI streaming surface) through an ASP.NET Core endpoint to the chosen transport: response flushing/buffering, content negotiation, and propagating **client-disconnect → `CancellationToken` → abort the upstream Anthropic call** so we stop paying for tokens nobody is reading.
4. **Persistence reconciliation.** The assistant turn is event-sourced. Recommend the persist-on-complete pattern: append **when** (after the full stream completes vs. incrementally), idempotency under retry/replay (the codebase uses an `IIdempotencyStore` keyed on client GUIDs), how the **user turn** is persisted relative to the assistant turn, and the **failure semantics if the stream dies mid-way** — discard, persist a partial+errored turn, or mark for retry? What does the client see on reload after a mid-stream death?
5. **Mid-stream safety + error UX.** Slice 3 enforces safety **pre-call** (a deterministic `SafetyGate` keyword classifier on the user input) and could judge **post-call**. Is **mid-stream deterministic red-line checking** (buffer to a sentence boundary, abort or splice a corrective clause on a banned token sequence) a real, recommended production pattern, or is the practical posture **pre-call gate + abort-only mid-stream + post-stream async judge**? On an LLM failure mid-stream, what is the recommended client UX (render the partial, surface a retry affordance, idempotent retry)?
6. **Auth + gotchas.** Confirm `CookieOrBearer` (`__Host-` cookie + antiforgery) works over the chosen transport via fetch-based streaming, and enumerate the gotchas that silently break token streaming: HTTP/2 vs HTTP/1.1, Kestrel/response-compression buffering, intermediary proxy buffering, and any interplay with the existing prompt-injection sanitization on user-influenced streamed content.

## Why It Matters

This is the load-bearing architecture for the heart of Slice 4. Getting it wrong yields one of three bad outcomes: a non-streaming chat (fails the explicit functional requirement), a bolt-on second data-fetching stack that fights RTK Query across the app, or a persistence model that silently loses turns on mid-stream failure. It is a near-one-way door at the foundation of the conversation surface — expensive to unwind once real conversation history exists — and it interacts with whether the per-plan `ConversationLogView` projection can hold open-conversation and off-plan turns or needs a user-scoped redesign. The requirements doc names streaming + RTK Query as the #1 risk; this artifact is the gate before the conversation-core spec can lock.

## Deliverables

- **Concrete end-to-end recommendation:** transport + RTK Query client pattern + server pipeline + persistence-reconciliation approach + mid-stream safety/error posture, presented as one coherent design a single developer can implement.
- **Alternatives considered and why rejected** — especially SSE vs. WebSocket vs. SignalR, and `onCacheEntryAdded` vs. raw-fetch-into-local-state.
- **Version pins** for the recommended pattern: Redux Toolkit / RTK Query, React 19, .NET 10 / ASP.NET Core, Microsoft.Extensions.AI, Anthropic SDK — including any minimum versions the pattern requires.
- **Gotchas & security implications:** auth-over-stream, Kestrel/proxy buffering, cancellation/backpressure, antiforgery, sanitization interplay with streamed output, and any security concern from streaming user-influenced content back token-by-token.
- **Persistence-model note:** state explicitly whether the recommended streaming/persistence pattern constrains or is neutral to the plan-scoped-vs-user-scoped conversation-stream decision, so that adjacent choice can be made deliberately in the conversation-core spec.
