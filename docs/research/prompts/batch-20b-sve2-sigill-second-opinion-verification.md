## Context

RunCoach's Slice 0 is about to merge the T02.6 backend dev-env work. The containerized Path A (`tilt up` → `docker compose build api` → `dotnet restore` inside the SDK image) is documented as broken on Apple Silicon per R-063 + DEC-056. Before shipping that documentation as the durable answer, the author wants a second-opinion pass to rule out:

1. **Incorrect env-var name**. R-063's `DOTNET_EnableArm64Sve` / `_Sve2` names came from the artifact's reading of `src/coreclr/inc/clrconfigvalues.h` but may not match the **actual runtime-accepted** name on SDK 10.0.203 / runtime 10.0.7. CoreCLR has historically accepted multiple prefixes (`DOTNET_`, `COMPlus_`, possibly `DOTNET_Jit…` for JIT-level knobs), and 10.x may have renamed or case-normalized the exact key. A wrong-name condition would explain both (a) why the Dockerfile `ENV` probe shows the var in `env | grep -i sve` but the JIT doesn't honor it, and (b) why the master switch was only probabilistically effective.
2. **Missing companion knob**. `Sve` / `Sve2` without also setting `Sme` / `Sme2` / `Sve2Bitperm` / `Sve2Sm4` / `Sve2Sha3` etc. may leave an ISA slice enabled that still faults. The R-063 probe did test `_Sme=0` + `_Sme2=0` alongside (5/5 FAIL) but the CoreCLR config namespace may expose sub-feature gates (`_Sve2Bitperm`, `_Sve2Aes`, `_Sve2Sm4`, `_Sve2Sha3`, `_Sve2I8mm`, etc.) that need independent zeroing.
3. **Runtime-config properties, not env vars**. The `runtimeconfig.template.json` / `runtimeconfig.json` `configProperties` block exposes a parallel config surface (`System.Runtime.Intrinsics.…`, `System.Numerics.…`). Setting properties at the JSON layer may override env vars on the `dotnet restore` tooling path whose host runtimeconfig the env vars do not reach.
4. **JIT-level disable flags**. Separate from the ISA enable/disable surface, RyuJIT exposes `DOTNET_JitDisasm`, `DOTNET_JitNoSve`, `DOTNET_JitNoSve2`, or similar. These may be the actual knobs the JIT consults, not the `EnableHWIntrinsic*` knobs the runtime exposes at higher layers.
5. **Alternative container runtime**. Colima 0.10.1 + containerd is one of three reasonable Mac options; **Docker Desktop** (VMM) and **OrbStack** (OrbStack-native VM) expose different CPU feature masks and different HWCAP surfaces. Either may already mask `HWCAP2_SVE2` out of the guest kernel's view, eliminating the bug without any app-side change. Community reports from 2026-Q1 on #122608 and adjacent issues should be surveyed.
6. **Alternative VM backend / CPU mask**. Colima is a thin wrapper over Lima. Lima `templates/default.yaml` + `cpuType: "host,+sve"` or `"host,-sve,-sve2,-sme,-sme2"` style QEMU `-cpu` flags may be tunable to mask SVE2 before the guest sees it. Same for Podman Machine (`podman machine init --cpu-type=…`).
7. **Preview / nightly SDK with the fix backported**. R-063 asserted "no 10.0.x servicing backport is planned" based on the `11.0.0` milestone. That assertion should be re-checked against the current `dotnet/installer` release branch, `dotnet/sdk` nightly feeds, and the `main` channel daily builds — Microsoft has backported JIT fixes before when the fix was low-risk. Specific images to test: `mcr.microsoft.com/dotnet/nightly/sdk:10.0`, `:10.0-preview`, `:10.0.204`, `:10.0.300`. Also check whether there's an aspnetcore / sdk prerelease channel tag.
8. **Cross-SDK build**. Using the .NET 9 SDK container (`mcr.microsoft.com/dotnet/sdk:9.0`) to restore and build a `net10.0` solution. SDK 9.x does not flip the Sve2 detection switch (pre-#115117), so the JIT bug cannot fire. Cross-targeting from SDK 9 → TFM `net10.0` has supply-chain constraints (reference assembly packs need to be available) but is mechanically a common pattern.
9. **Host-side restore + COPY strategy**. Skip `RUN dotnet restore` inside the container entirely: run `dotnet restore` on the host, bind-mount the populated `~/.nuget/packages` into the build stage, run `dotnet publish --no-restore`. The host JIT is fine. Whether this pattern is robust across arm64 host + arm64 container (no RID mismatch) is the sub-question.
10. **NativeAOT / ReadyToRun angle**. Publishing with `<PublishAot>true</PublishAot>` or `<PublishReadyToRun>true</PublishReadyToRun>` changes the JIT engagement surface. Whether it eliminates the restore-phase tier-up that triggers the SIGILL is worth checking, even if the operational cost is high.
11. **Exact same shell invocation pattern**. R-063's empirical tests used `docker run -e DOTNET_EnableHWIntrinsic=0` AND Dockerfile `ENV` — both failed 5/5 except for one lucky `-e` run. Is there a Dockerfile pattern where the env reaches the JIT deterministically (e.g., `RUN env DOTNET_EnableHWIntrinsic=0 dotnet restore …` inline, rather than relying on inherited `ENV`)? Subtle shell / sh / bash / dotnet host startup order may matter.
12. **Community-discovered patches that post-date R-063's search window**. #122608 is active (opened 2025-12-17, still open). Daily updates on the thread + adjacent threads (#112605, #119174, #109652) + the `arm-sve` label queries should be re-scanned as-of the research date. Cross-ecosystem (JVM #631, podman #28312) may have posted new workarounds that translate to CoreCLR.

None of these are hypothetical comfort items — the author wants each category checked rigorously before accepting "Path A broken until .NET 11 GA." The empirical reference hardware is an **Apple M4 Pro on macOS Darwin 25.4.0** running **Colima 0.10.1 default profile aarch64 4 CPU / 8 GiB**, with Docker server 27.4.0 and SDK image digest `sha256:8a90a473da5205a16979de99d2fc20975e922c68304f5c79d564e666dc3982fc` (SDK 10.0.203 / runtime 10.0.7, pushed 2026-04-21).

## Research Question

**Primary:** Is there any deterministic (100% reliable across ≥10 consecutive runs) workaround that makes `dotnet restore RunCoach.slnx` succeed inside the `.NET 10 SDK 10.0.203` container on the reference Apple M4 Pro + Colima hardware, without switching to Rosetta and without waiting for .NET 11 GA? If yes, produce the exact Dockerfile + repository diff and a verification procedure. If no, confirm with primary-source citations that R-063's demote-Path-A posture is correct.

**Sub-questions:**

1. **Exact env-var / config-key name.** Inspect `dotnet/runtime` `release/10.0` branch at `src/coreclr/inc/clrconfigvalues.h` AND `src/coreclr/jit/jitconfigvalues.h` AND `src/coreclr/vm/hwintrinsic.cpp`. What are the **exact** accepted names for disabling SVE, SVE2, SME, SME2, and each sub-feature? Verify against the `CLRConfig::GetConfig…` call sites that parse env vars. Do they accept `DOTNET_` only, or also `COMPlus_` / `DOTNET_Jit…`? Are names case-sensitive or normalized?

## Why It Matters

- **Blocks the last validation of the T02.6 PR.** Slice 0 T02.6 ships the Tilt + containerized-api + Swagger workflow. If `tilt up` is permanently broken on the author's primary hardware until .NET 11 ships (expected Q4 2026, ~6 months out), the PR ships with a documented-broken feature. Every future contributor on M3/M4/M5 hits this the first time they follow `CONTRIBUTING.md`.
- **Rejecting a workaround that exists is the expensive error.** R-063's strategic recommendation was "fix Path A durably + demote." We collapsed to "demote only" because the primary fix didn't hold empirically. But R-063's diagnosis was right; its empirical hit-rate was one mechanism (ENV). There may be a different mechanism (runtime-config property, alternative runtime, sub-feature knob, host-side restore) that is deterministic. Accepting "it's broken" without exhausting the option space costs six months of contributor friction.
- **Accepting a workaround that doesn't exist is the reversible error.** If this research confirms R-063 + DEC-056, we lose one research round (maybe four hours) but strengthen the confidence in the demotion posture with a second primary-source citation. That's worth four hours.
- **Supply-chain posture.** Some workaround candidates (disabling NuGet signature verification, downgrading SDK digests, switching to nightly tags) weaken the supply chain. Any workaround this research recommends must preserve signature verification + be pinned to a stable / servicing release, not a nightly tag that floats.
- **The research agent should be empirical.** R-063's failure mode was "the primary source said X, X failed in practice on the target hardware." The verification script `scripts/repro-sigill.sh` already landed in R-063 §3. This research should include executable test procedures so the author can confirm or refute each candidate workaround on their machine without re-doing the literature review.
