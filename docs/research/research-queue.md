# Research Queue

> **User-facing rename (2026-04-15).** Historical research topic names and body paragraphs in this file preserve the original term "VDOT" as used at the time each research topic was raised. Per DEC-043, the project's user-facing surface now uses "Daniels-Gilbert zones" or "pace-zone index" terminology to avoid exposure to The Run SMART Project LLC trademark (Runalyze enforcement precedent). Future research topics added to this queue should use the new terminology unless the topic is specifically about the trademarked term itself.

Topics to hand off to a deep research agent. Each entry captures the question, why it matters, and where findings should feed back into the planning docs. Full research artifacts live in `research/artifacts/` — only outcomes and key takeaways get pulled into the main planning documents.

## How to Use

1. Add a topic below with a clear research question and context on why it matters.
2. Hand off to a deep research agent.
3. Store the full output in `artifacts/` (e.g., `artifacts/competitive-landscape.md`).
4. Summarize key takeaways back into the relevant planning doc and update status here.

### Batch 15 (MVP-0 Slice 0 foundation pattern lock-in — pre-implementation research)

- **Batch 15:** R-044 + R-045 + R-046 + R-047 — four-artifact pre-implementation pass that drove a meaningful redesign of the Slice 0 spec. The headline finding is **R-044's reframe**: don't put a JWT in the browser at all. The Slice 0 spec was rewritten on integration to adopt this and the three companion findings. Summary of integrated takeaways:

  - **R-044 (browser auth)** → **ASP.NET Core Identity application cookie** (`__Host-` prefix, `HttpOnly`, `Secure`, `SameSite=Lax`) is the browser session; **no JWT issued to the browser** in MVP-0. Anti-CSRF via ASP.NET Core's built-in antiforgery middleware (`X-XSRF-TOKEN` header echoing a `XSRF-TOKEN` cookie). Refresh-survival via Identity sliding expiration — no `/refresh` endpoint for the SPA. `AddJwtBearer` is registered as a non-default opt-in scheme reserved for the future iOS shim; a `CookieOrBearer` authorization policy is wired on day one so iOS lands additively. **Mount a Docker volume for `/keys` so DataProtection survives container rebuilds** — single biggest operational foot-gun in the recommendation.

  - **R-045 (auth endpoints)** → **Custom `AuthController` in `Modules/Identity/`**, NOT `MapIdentityApi<TUser>()`. Four hard limits of `MapIdentityApi` collide with the project: it issues an opaque DataProtection blob (not a JWT, blocking the iOS-bearer future), the `RegisterRequest` is sealed (blocks atomic `UserProfile` creation in Slice 1), endpoints can't be selectively excluded (`/refresh`, `/forgotPassword`, etc. always mount), and `/manage/info ≠ /me` (returns only email + isEmailConfirmed). `dotnet/eShop` and David Fowler ("not ideal") confirm. R-045's wiring sketch was written assuming bearer; on integration it was reshaped to issue the cookie via `SignInManager.PasswordSignInAsync` (R-044's pattern) while preserving the custom-controller architecture and preserving room for a `POST /api/auth/login-bearer` endpoint when the iOS shim lands.

  - **R-046 (migrations + tests)** → **EF Core 9+ `IMigrationsDatabaseLock`** kills the historic objection to `Database.MigrateAsync()` on startup; recommendation is dev/CI = `MigrateAsync` in fixture / startup, **production = `dotnet ef migrations bundle` from CD** with two Postgres roles (`runcoach_migrator` for DDL, `runcoach_app` for DML). Test fixture: **`[assembly: AssemblyFixture(typeof(RunCoachAppFactory))]`** (xUnit v3's new primitive — supersedes per-class `IClassFixture` for DB-backed suites), single Postgres container per assembly, **Respawn on `public` schema only** + **Marten's `Host.ResetAllMartenDataAsync()`** on the events schema. Container reuse on locally only; `MartenDaemonModeIsSolo()` forced in tests to avoid advisory-lock contention.

  - **R-047 (Marten)** → **Marten 8.28 + Wolverine 5.28 + Npgsql 9** as the pin. Schema name **`runcoach_events`** (not the placeholder `marten`) per JasperFx convention. `StreamIdentity.AsGuid` with **stream id = `UserId`** (one Plan per user). `TenancyStyle.Conjoined` with `tenant_id = userId.ToString()`. **`SnapshotLifecycle.Inline`** single-stream projection; reads dominate so Inline + `Marten.AspNetCore`'s zero-copy `WriteLatest<Plan>(userId, http)` is the right shape. Async daemon `DaemonMode.Solo` in MVP-0; flip to `UseWolverineManagedEventSubscriptionDistribution = true` at MVP-1+. Cross-store consistency via Wolverine outbox composing `IntegrateWithWolverine()` (Marten) + `AddDbContextWithWolverineIntegration<AppDbContext>()` (EF) + `AutoApplyTransactions()`. **One live risk**: Marten 8.28 lists .NET 10 as "untested" — pin Marten ≥ 8.20, monitor JasperFx repo, fall back to test-assembly-only `net9.0` if a hard blocker surfaces.

  Together, these four artifacts triggered (a) a major Slice 0 spec rewrite (token-storage abstraction removed; cookie + antiforgery + dual-scheme architecture written in; Marten registration shape locked; AssemblyFixture pattern locked; Marten schema renamed `runcoach_events`) and (b) a new decision-log entry capturing the cookie-not-JWT pivot.
