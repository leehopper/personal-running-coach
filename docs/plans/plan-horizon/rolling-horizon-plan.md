## 1. Problem

**The micro tier is broken from day 8 of every plan, today, in production.** `PlanGenerationService.GeneratePlanAsync` emits exactly one micro-detail event — `FirstMicroCycleCreated` — hardcoded to week 1 (`FirstMicroCycleCreated.cs`; `PlanProjection.Apply(FirstMicroCycleCreated, dto)` writes it to `MicroWorkoutsByWeek[1]` unconditionally, `PlanProjection.cs:142-149`). `WorkoutLogService.ResolvePrescriptionAsync` resolves a run's prescription via `plan.MicroWorkoutsByWeek.TryGetValue(slot.WeekNumber, out var micro)` (`WorkoutLogService.cs:190`) — for any week but 1, that lookup misses and the run silently resolves off-plan (`WorkoutLogService.cs:183-213`, confirmed by direct read). **Every plan is functionally dead from the second week onward: weeks 2-4 have meso day-slots (Run/Rest/CrossTrain) but zero prescribed sessions to log against.** This is not a week-5 edge case — it is live for every runner in week 2, i.e. day 8.

**The meso tier hits a hard wall at week 4, by design, with no path past it.** `PlanEventSequence` validates the canonical shape `[PlanGenerated, MesoCycleCreated×4, FirstMicroCycleCreated]` via a hard equality guard, `Mesos.Count == ExpectedMesoCount` where `ExpectedMesoCount = 4` (`PlanEventSequence.cs:33,69`), and `PlanGenerationService.MesoWeekCount` is a hardcoded `internal const int = 4` driving the generation loop (`PlanGenerationService.cs:136,143,341-343`). Both constants govern only the two `StartStream` call sites — `SubmitStructuredAnswersHandler` (onboarding) and `RegeneratePlanHandler` — but nothing else in the codebase ever appends meso week 5+ to an existing stream.

**Adaptation cannot fix either gap — it only edits, never creates.** `PlanProjection.Apply(PlanAdaptedFromLog, dto)` revises `WeeklyTargetKm` on an existing `MesoWeeks` entry (`FindIndex` + in-place replace, `PlanProjection.cs:198-201`) and swaps workouts on an existing `MicroWorkoutsByWeek` entry (`TryGetValue` + in-place replace, `PlanProjection.cs:228-231`). Both apply methods explicitly skip — never synthesize — a week absent from the projection (`PlanProjection.cs:204-206,224-227`, doc comments confirm this is deliberate: "the projection never synthesizes weeks the plan-generation flow did not emit"). A plan of any declared length beyond 4 weeks has no code path, anywhere, that ever populates week 5+.

**The frontend already knows and is quietly hiding it.** `use-plan.hooks.ts`'s `resolveCurrentWeek` clamps the date-derived current week into `[firstWeek, lastWeek]` of whatever weeks currently carry a meso template — today `lastWeek` is permanently 4, so every plan past week 4 silently renders stale week-4 data forever, with zero visual signal anything is wrong. **That clamp is correct and stays** — it is the right defensive fallback for staleness (a slow sweep tick, an outage). What changes is what it clamps *against*: today `lastWeek` never moves past 4; once this ships it grows as real weeks are populated, so in steady state the clamp converges to the runner's actual week and only visibly engages during the bounded window between crossing into a new week and the next sweep tick.

## 2. Goal & Done-State

**Goal.** A plan of any declared length has real meso *and* micro detail for the runner's current week, maintained some buffer ahead of them, for as long as the runner keeps training — without ever touching a user-facing request.

**Hard invariant — no LLM call ever runs inside a user-facing request.** Onboarding completion, `POST /plan/regenerate`, `POST` workout-log create, and `GET /plan/current` are all completely unmodified in their request-time behavior by this work. Every new LLM call this design adds happens inside an isolated `BackgroundService` tick that can fail, retry, or circuit-break without ever becoming a 500 or a slow page load for a real request. This is a load-bearing invariant, not a preference — it is what makes this fixable without touching, stalling, or being stalled by the in-flight SPLIT UI redesign cycle.

**Done-state.**

- Every active plan self-extends: the lowest week still missing meso or micro detail, within a bounded buffer of the runner's actual calendar week, gets populated automatically — including every plan already live today (see D3; this is not a migration, it is the steady-state behavior applied on the first tick).
- A plan's rolling extension is planned from real signal — adherence, current adaptation-signal state, restructure count, the already-adapted current weekly target — not just the frozen onboarding-day snapshot (see D6).
- Onboarding, regenerate, log-create, and plan-read stay byte-for-byte unmodified in behavior and latency.
- The existing 5-profile eval fixture suite (`plan.*.mesoweek`/`micro`) stays byte-identical — zero re-record, zero DEC-074 trigger, zero disruption to the production prompt-prefix cache.
- The frontend's `resolveCurrentWeek` clamp needs zero code changes — it just gets real data to clamp against.

## 3. Locked Design Decisions

Base design: **sweeper-driven rolling horizon**, grafted with a deterministic `TrainingHistorySummary` feeding the extension prompt. Not open for re-litigation — see the design brief for the full three-design, three-lens adversarial comparison. Each decision below states its rejected alternative in one line; where a judge panel found a **fatal flaw** in that alternative, it's named explicitly so the reasoning isn't lost.

- **D1 — Trigger: a timer-only `BackgroundService`, never a user request.** New `PlanHorizonSweeper : BackgroundService` (`Modules/Training/Plan/Horizon/`), mirroring `IdempotencySweeper`'s `ExecuteAsync`/`SweepAsync` loop shape exactly (`Infrastructure/Idempotency/IdempotencySweeper.cs:98-129`): run immediately on host boot, `await SweepAsync(ct)` then `await Task.Delay(interval, timeProvider, ct)`, so ticks never overlap by construction. Registered via `services.AddHostedService<PlanHorizonSweeper>()` alongside the existing `AddHostedService<IdempotencySweeper>()` (`Infrastructure/ServiceCollectionExtensions.cs:147`). **Rejected: per-log dispatch off `WorkoutLogsController.CreateLog`** (mirroring `AdaptationEvaluationDispatcher`) — fatal flaw (judge panel, Correctness lens): the dispatcher's catch clause, copied "exactly" from `AdaptationEvaluationDispatcher.cs:28-29`, only narrows `EventStreamUnexpectedMaxEventIdException`/`DocumentAlreadyExistsException`; with no surrounding try/catch in the controller and the meso/micro validator optional rather than required, a horizon-extension LLM failure 500s the entire log-create request — discarding an already-committed log write. A second, independent fatal flaw: a per-log trigger gated to `currentWeek + 1` structurally starves any runner whose training days never land inside the gate window (e.g. a Thu-Sat micro lead-time gate never fires for a Sun/Mon/Tue training pattern) — permanently, not as a rare edge case.
- **D2 — Cadence and buffer: 30-minute sweep interval, 2-week populated buffer.** `PlanHorizonSettings.SweepIntervalMinutes = 30` (matches `IdempotencySweeper.SweepInterval`'s order of magnitude, tuned tighter since staleness here is user-visible, not just cleanup hygiene). `PlanHorizonSettings.BufferWeeks = 2`: keep meso *and* micro fully populated 2 weeks ahead of the runner's date-derived current week (`floor((today − PlanStartDate)/7) + 1`, the same arithmetic `PlanCalendar.ResolveSlot` already uses, `PlanCalendar.cs:71-72`), re-evaluated fresh every tick so a plan left idle for a month self-corrects on the next tick rather than drifting. **Rejected: 1-week buffer** — a single delayed tick (API hiccup, a validator rejection waiting for the next retry) could let the runner catch up to an ungenerated week. **Rejected: 4-week buffer** — generates weeks a mid-plan regenerate would discard before they're ever reached, and widens the "blind future week" gap the D6 graft exists to narrow. 2 weeks is the only option with an explicit argument: one sweep-interval-plus-retry-cycle of slack.
- **D3 — Selection: lowest incomplete week, not `currentWeek + 1`.** Each tick computes a pure `SelectNextIncompleteWeek(PlanProjectionDto, today, bufferWeeks) -> int?` — the lowest week index missing meso OR micro, capped at `min(currentWeek + BufferWeeks, macro.TotalWeeks)`. This is the mechanism that makes the "existing plans" question a non-issue: **every plan live today has `MesoWeeks = {1,2,3,4}` but `MicroWorkoutsByWeek = {1}`, so the lowest incomplete week for every one of them is week 2** — the sweeper backfills the live micro-tier bug (§1) as the ordinary first thing it does on its very first tick, one week per tick, before ever reaching week 5+. This is the same code path as steady-state extension, not special migration machinery — there is nothing to design here beyond the selection rule itself. **Rejected: `currentWeek + 1` only** (the JIT design's mechanism) — fatal flaw (judge panel, both Correctness and Architectural-fit lenses): it cannot backfill a multi-week gap (a plan stuck at week 4 while the runner is actually on week 9 generates week 10 on the next log, permanently skipping weeks 5-9) and cannot recover from a single missed logging window (once `currentWeek` advances past an ungenerated week, nothing ever re-targets it).
- **D4 — Contract: one new event, one new `Apply` overload, `PlanEventSequence` untouched.** New `MicroCycleCreated(int WeekIndex, MicroWorkoutListOutput Micro)` in `Modules/Training/Plan/`, structurally mirroring `MesoCycleCreated`. `FirstMicroCycleCreated` is **not** deleted or repurposed — Marten streams are append-only, and every plan already live carries it for week 1 forever; it keeps arriving exactly as today. New `PlanProjection.Apply(MicroCycleCreated, PlanProjectionDto)` overload generalizes the existing `Apply(FirstMicroCycleCreated, dto)`'s hardcoded `map[1] = ...` (`PlanProjection.cs:144-147`) to `map[@event.WeekIndex] = ...`, same defensive `WeekIndex >= 1` guard shape as `Apply(MesoCycleCreated, ...)`. `MartenConfiguration.RegisteredEventTypes` (`Infrastructure/MartenConfiguration.cs:60-78`) gets `typeof(MicroCycleCreated)` added — required, or `MartenStoreOptionsCompositionTests` fails by design. **`PlanEventSequence.ExpectedMesoCount == 4` is deliberately NOT relaxed.** It stays scoped exactly to the two `StartStream` call sites (onboarding, regenerate) it already governs — extension never constructs a `PlanEventSequence` at all; it appends `MesoCycleCreated`/`MicroCycleCreated` directly via `session.Events.Append` against the plan's existing stream. **Rejected: relax `PlanEventSequence` to a non-empty-count guard and generate all N weeks upfront in one call** (the Eager Full-Horizon design) — fatal flaw (judge panel, all three lenses): its own flagship correctness claim (tightening `WorkoutLogService.cs:185`'s `ResolveSlot` bound) is a verified no-op — the `MicroWorkoutsByWeek.TryGetValue` miss at line 190 already returns null identically regardless of that bound; two existing integration test files (`AdaptationOrchestrationIntegrationTests.cs`, `ConfirmConversationalLogEndpointIntegrationTests.cs`) construct `PlanEventSequence` via `with { Micro = new FirstMicroCycleCreated(...) }` and would fail to compile, undisclosed in that design's own test-blast-radius tally; and it commits up to 20 weeks of exact workout prescriptions on day 1 with zero training-history signal — the "static plan generator... that never adjusts" anti-pattern `vision-and-principles.md` opens by rejecting, at 5x today's scale.
- **D5 — Idempotency and concurrency: projection state is the source of truth; a marker is a disposable fast-path only.** Two deliberately redundant layers. (1) A durable pre-check at the start of every handler run: `dto.MesoWeeks.Any(w => w.WeekNumber == target)` / `dto.MicroWorkoutsByWeek.ContainsKey(target)` computed directly off the live projection — cheap, has no expiry, and is what actually prevents duplicate or regressive work. (2) A deterministic `IIdempotencyStore` marker (`Guid.CreateVersion5(PlanHorizonNamespace, $"{planId}:{targetWeekIndex}")`, insert-only, recorded in the same transaction as the event append) as a fast-path only — **not** the source of truth, because `IdempotencySweeper` deletes *any* marker older than 48h regardless of purpose (`IdempotencySweeper.cs:31`, `IdempotencyMarker`'s single global `RecordedAt`), and this handler re-asks "did we already do week N" every 30 minutes across a plan's multi-month life — a marker alone would silently stop being trustworthy mid-plan. Concurrency: `ExtendPlanHorizonHandler` registers its own `Configure(HandlerChain)`/failure policy — `OnException<EventStreamUnexpectedMaxEventIdException>().RetryTimes(3).Then.MoveToErrorQueue()` — mirroring `EvaluateAdaptationHandler`'s exactly (`Modules/Training/Adaptation/EvaluateAdaptationHandler.cs:134`, retry rule ~`:383-385`), because this handler appends to the same Marten stream `EvaluateAdaptationHandler` already writes to, and the two will race at this scale (a runner logging a workout at the moment the sweeper ticks for their plan). A full retry re-runs the handler body, so the idempotency pre-check re-executes against fresh state — no torn or partial appends. Per-plan failure isolation: each plan's dispatch inside a sweep tick gets its own try/catch, so one plan's terminal rejection never aborts the tick for others (a strengthening over `IdempotencySweeper`'s precedent, which wraps the whole tick — `IdempotencySweeper.cs:104-118` — not per-item; there is no per-item catch to copy, this is new). A new cross-tick circuit breaker (`PlanHorizonExtensionState` — plain, non-event-sourced Marten document, `[Identity] PlanId`, `ConsecutiveFailureCount`, `LastAttemptedWeekIndex`, `LastFailureAt`/`Reason` — mirroring `AdaptationSignalStateDocument`'s plain-document bookkeeping role) trips after `MaxConsecutiveFailures = 5` and stops retrying that plan until a human looks. **Rejected: port `EvaluateAdaptationCommand`'s marker-only idempotency pattern verbatim** — that pattern only needs to answer "did we process this log, ever" once, near creation; this question gets re-asked for months, which the 48h retention window structurally cannot support alone.
- **D6 — The `TrainingHistorySummary` graft: deterministic, non-LLM, suffix-only.** A new `TrainingHistorySummarizer` (`Modules/Training/Plan/Horizon/`) computes a small structured record — `WeeksCompleted`, `AdherenceRatePercent` (completed-vs-prescribed ratio over existing `WorkoutLog` data), `CurrentSignalState` (the plain `AdaptationSignalState` already persisted via `AdaptationSignalStateDocument`, reused verbatim), `RestructureCount` (a plain count of restructure events on the stream so far), `MostRecentWeeklyTargetKm` (the already-adapted, ground-truth `WeeklyTargetKm` off the most recently completed `MesoWeeks` entry — reflecting any prior restructure, not macro's static day-1 guess) — and threads it into the horizon-extension prompt. **Why this must be deterministic, not LLM-derived:** root `CLAUDE.md`'s Architecture Principles are explicit — "never use LLMs for structured data tasks (pace calculations, zone math, ACWR)... LLMs handle coaching conversation, plan narrative, and adaptation reasoning." Adherence rate and restructure count are exact arithmetic over data the system already has; computing them via a model call would be non-reproducible, slower, and a direct violation of the deterministic-computation/LLM-coaching split that is the project's first architectural law. The summary is injected via **new, separate sibling builder methods** (`BuildHorizonExtensionMesoUserMessage`/`...Micro...` on `PlanGenerationService`) that never edit `BuildMesoUserMessage`/`BuildMicroUserMessage` and never touch `IContextAssembler.ComposeForPlanGenerationAsync`'s shared cacheable prefix — a suffix-only addition, rendered only for extension calls, so the original bootstrap chain's rendered bytes for weeks 1-4 are untouched and none of the 5 existing eval fixtures or the production Ephemeral1h prompt-prefix cache are disturbed. **Rejected: ship the base Sweeper design without this graft** (context-blind extension, reusing `ComposeForPlanGenerationAsync`'s prefix as-is) — not fatal, but it's the one place the judge panels agreed the base Sweeper design underperforms: it implements "guess now via a timer, correct later via the separate adaptation loop" rather than "reason from real history at generation time," which `vision-and-principles.md`'s stated product thesis (line 13: the coaching intelligence "reasons about [training history] continuously") specifically calls for. **Rejected: thread history into the shared prefix instead of a suffix** — busts all 5 existing fixtures and the production prompt cache for a gain available more cheaply via a suffix.
- **D7 — Failure handling: no synchronous fallback, no new user-facing error surface, existing clamp absorbs staleness.** By construction of D1, extension failure can never turn into a request-path error — the worst state this design can produce for any plan that finished initial generation is "behind schedule" (a future week not yet populated), never "broken," because `PlanEventSequence`'s all-or-nothing guard (untouched, D4) means a plan stream is either fully absent or already carries 4 complete meso weeks; there is no code path, before or after this change, that leaves a plan with zero populated weeks. The existing frontend clamp (`resolveCurrentWeek`, unchanged — §1) keeps the plan non-blank throughout any staleness window. The circuit breaker's only externally visible signal for this ship is a structured log line and a new OTel metric (`runcoach.plan.horizon.extend.rejected`) — no alerting/dashboard is built. In a solo-dev, no-external-paging environment this is a named, accepted residual risk (a systemic failure — bad API key, model deprecation, sustained outage — could leave the builder's own plan silently stale until a human happens to look), not a solved one. **Rejected: surface a user-visible "coach still planning this week" vs. "genuinely off-plan" distinction now** — no wire field exists for it and adding one is exactly the kind of new user-facing surface this ship's invariant (D-goal) is designed to avoid; named as a natural, small follow-up, not built here.
- **D8 — `MesoWeekOutputValidator` scope: required on the extension path, not backported.** A new deterministic validator trio (`MesoWeekOutputValidator`/`...ValidationResult`/`...ValidationViolation`, mirroring `MacroPlanOutputValidator`'s `Valid()`/`Invalid()` factory shape) checks `Meso.WeekNumber == expectedWeekIndex` and day-slot completeness before ever staging a new `MesoCycleCreated` for an extension week — required, not optional, because `PlanProjection.Apply(MesoCycleCreated, ...)`'s hard `InvalidOperationException` guard on a `WeekNumber` mismatch (`PlanProjection.cs:111-115`) is real and load-bearing: an unvalidated mismatch there hard-aborts the whole extension transaction. This closes a gap the original 4-week bootstrap loop has always had — `GeneratePlanAsync`'s existing meso loop performs **zero** meso validation today (confirmed: no `MesoWeekOutputValidator.cs` exists anywhere in the repo before this work). **Locked per the builder: backporting this validator to `GeneratePlanAsync`'s original bootstrap loop is explicitly OUT of this ship — filed as a fast-follow** (see § Scope: Out), not because it isn't worth doing (all three judge panels flagged it as real and cheap), but because reopening an already-stable, already-tested path isn't required to ship the rolling horizon itself.

## 4. Functional Requirements

- Any active plan, of any declared length, has real meso and micro detail for the runner's current week and the next `BufferWeeks` weeks, maintained automatically as the runner progresses — no manual regenerate required to "catch up."
- Every plan already live today self-heals on its first sweep tick after deploy, via the ordinary D3 selection rule — no separate migration step exists or is needed.
- Extension never proceeds past `macro.TotalWeeks`, for anchored and NoAnchor plans alike (both always carry a concrete `MacroPlanOutput.TotalWeeks`, even though `PlanHorizon.NoAnchor()` itself carries no `TargetTotalWeeks` — see `PlanHorizon.cs:11-40`).
- A regenerated plan (fresh stream, fresh `PlanStartDate`) is unaffected by any prior stream's extension state; the sweeper re-derives `currentWeek` and gaps against the new stream on its own next tick, with no special-casing required.
- Onboarding completion, `POST /plan/regenerate`, `POST` workout-log create, and `GET /plan/current` are unmodified in request-time behavior, latency, and error surface.
- A runner who never logs a workout still gets their plan extended — the trigger is calendar-only, never gated on any user action.
- The extension prompt for a given week reflects `TrainingHistorySummary` (D6) rather than only the frozen onboarding-day snapshot.
- The existing 5-profile eval suite (`plan.{sarah,lee,maria,james,priya}.mesoweek`/micro fixtures) renders byte-identical output for the unchanged bootstrap path; zero fixture re-record.

## 5. Quality Requirements

- Idempotency: re-running the sweeper or the handler for a week that's already fully populated is a correct, cheap no-op (one document load).
- Concurrency: `EvaluateAdaptationHandler` and `ExtendPlanHorizonHandler` racing on the same plan stream converge without torn state via the existing optimistic-concurrency-plus-chain-scoped-retry idiom (D5).
- Failure isolation: one plan's LLM or validator failure never aborts a sweep tick for any other plan.
- Cost ceiling: total extension spend for any single plan is bounded by `macro.TotalWeeks × 2` calls and never exceeds the plan's own declared length; a runner who stops training triggers zero further spend past their last-reached week plus the buffer.
- Observability: new OTel spans/metrics mirroring `PlanGenerationService`'s existing instrumentation pattern; every circuit-breaker trip is warn-logged.
- Test coverage follows `backend/CLAUDE.md`'s Wolverine handler integration-coverage rule — drive `ExtendPlanHorizonHandler` via `IMessageBus.InvokeAsync`/`InvokeForTenantAsync` against the real bus, never call `Handler.Handle` directly in test code.
- `MartenStoreOptionsCompositionTests` passes with the new `MicroCycleCreated` registration.
- New eval coverage recorded via the repo's documented targeted single-scenario recipe (`EVAL_CACHE_MODE=Record`, patched fixture expirations, `EVAL_CACHE_MODE=Replay` to verify) — no `.prompt-hashes.sha256` touch, since no prompt YAML changes.

## 6. Scope: In

- `PlanHorizonSweeper` (`BackgroundService`, 30-min cadence, 2-week buffer, lowest-incomplete-week selection via `SelectNextIncompleteWeek`, per-plan try/catch isolation, circuit-breaker check/update).
- `ExtendPlanHorizonCommand` + `ExtendPlanHorizonHandler`, `PlanHorizonExtensionState` document, dual idempotency (projection-state source of truth + disposable marker fast-path), chain-scoped `EventStreamUnexpectedMaxEventIdException` retry rule.
- `MicroCycleCreated` event, `PlanProjection.Apply(MicroCycleCreated, ...)` overload, `MartenConfiguration` registration.
- `MesoWeekOutputValidator` trio — required on the new extension path.
- `TrainingHistorySummary` + `TrainingHistorySummarizer` (deterministic) and its wiring into new sibling suffix-only prompt builders.
- `PlanHorizonSettings` (`BufferWeeks`, `SweepIntervalMinutes`, `MaxConsecutiveFailures`) + DI wiring.
- The natural, zero-extra-machinery backfill of every plan currently live (a consequence of D3, not a separate deliverable).
- New unit/integration/eval test coverage per § Quality Requirements.

## 7. Scope: Out (Deferred)

- **`MesoWeekOutputValidator` backport to `GeneratePlanAsync`'s original 4-week bootstrap loop.** Real, cheap, and flagged by every judge panel as worth doing — filed as a **fast-follow**, deliberately not bundled into this ship (locked by the builder; do not re-open).
- **NoAnchor (general-fitness, no race date) plan rollover past the last declared week.** A genuine open question — sent to research, not decided here (see § 8).
- Alerting/dashboard on the `PlanHorizonExtensionState` circuit breaker — a structured log line only for this ship.
- An optional `HorizonStatus` field surfaced on `PlanRenderingController`'s response (would enable a future "plan still updating" frontend banner) — not required for this design's core reliability property and not built.
- Parallel/batched dispatch across many plans within one sweep tick — sequential per-plan processing is a deliberate MVP-0-scale simplicity choice; revisit if active-plan count grows past the low tens.
- Any synchronous, request-path fallback for a stale/behind-schedule plan — the existing frontend clamp is the only user-visible degrade for this ship.

## 8. Open Unknowns

**NoAnchor plan rollover is a genuine Research Protocol trigger and is not decided in this document.** Once the sweeper is filling weeks up to whatever `TotalWeeks` the macro tier declared, a `PlanHorizon.NoAnchor()` plan (general-fitness, no race date) eventually reaches its last declared week with the runner still training. What happens next — re-run the macro tier for a fresh block, extend `TotalWeeks` in place, or cap the rolling window regardless of `TotalWeeks` — is a multi-option question with non-obvious tradeoffs and no existing precedent in this codebase. Per root `CLAUDE.md`'s Research Protocol, this must not be guessed at by an implementing agent.

- **Prompt:** `docs/research/prompts/plan-horizon-noanchor-rollover.md` (written).
- **Queue entry:** R-087 in `docs/research/research-queue.md` (queued).
- **Placeholder for this ship:** `WeekContext.FromMacro`'s existing defensive fallback — when the requested week index exceeds the macro's declared phase sum, it returns the last declared phase unchanged (`WeekContext.cs:64-66`) — already covers this case at zero additional code, for anchored and NoAnchor plans alike. This degrades gracefully (the extension keeps producing plausible-shaped weeks) but not scientifically (a runner rolling past week 20+ trains indefinitely on stale phase parameters). It is an explicitly-named, accepted placeholder, not a decision about what *should* happen — the research artifact, once integrated, replaces it.

## 9. PR Sketch

The design brief proposes ~4-5 PRs; this plan keeps 5 for review-size reasons but names the compression option. Each PR should carry its own fresh spec at build time.

1. **PR1 — Event, projection, and validator plumbing.** `MicroCycleCreated` event; `PlanProjection.Apply(MicroCycleCreated, ...)`; `MartenConfiguration` registration; `MesoWeekOutputValidator` trio; new sibling prompt-builder methods (`BuildHorizonExtensionMesoUserMessage`/`...Micro...`) + `GenerateWeekAsync` on `IPlanGenerationService`/`PlanGenerationService` (needs-meso/needs-micro conditional generation, reusing the existing bounded-retry-with-corrective-hint shape).
2. **PR2 — The handler.** `ExtendPlanHorizonCommand`/`ExtendPlanHorizonHandler`; `PlanHorizonExtensionState` document; dual idempotency (marker fast-path + projection-state source of truth); chain-scoped `Configure(HandlerChain)` retry rule.
3. **PR3 — The sweeper.** `PlanHorizonSweeper` `BackgroundService`; `PlanHorizonSettings`; DI wiring (`AddHostedService`); `SelectNextIncompleteWeek` helper; circuit-breaker check/update. This is the PR that makes the NoAnchor rollover question live; the research prompt for it (§8) is already written and queued as R-087.
4. **PR4 — The graft.** `TrainingHistorySummary` + `TrainingHistorySummarizer` (deterministic); wiring into the extension suffix builders from PR1.
5. **PR5 — Tests and eval.** Unit coverage (`SelectNextIncompleteWeek`, `MesoWeekOutputValidator`, `TrainingHistorySummarizer`, new `PlanProjection.Apply` overload); integration coverage for `ExtendPlanHorizonHandler` via the live bus (no-op path, meso-then-micro generation path, append-race-retry path); a `WolverineErrorRoutingTests`-style addition for the new chain-scoped rule; a new `PlanHorizonExtensionEvalTests` class (2-3 targeted, additive, funded-key-recorded scenarios: fresh-week extension, micro-only backfill against an already-adapted meso week, a deep-week phase-fallback case).

**Compressible to 3 PRs** (merge PR1+2, PR3+4) if fewer review cycles are preferred — a style call, does not change what ships.

## 10. Sequencing with the SPLIT Cycle

**Parallel, not fold-in, not wait — verified, not assumed.** This design's backend work lives entirely under a new `Modules/Training/Plan/Horizon/` folder plus surgical, additive edits to `PlanProjection.cs`, `Infrastructure/MartenConfiguration.cs`, and `PlanGenerationService.cs`/`IPlanGenerationService.cs`. None of the SPLIT UI cycle's slices open any of those files. `PlanProjectionDto` (`Models/PlanProjectionDto.cs`) is untouched — `MicroWorkoutsByWeek` is already `IReadOnlyDictionary<int, MicroWorkoutListOutput>`, generically keyed — so there is **no OpenAPI/wire-contract change and no `npm run codegen` churn**: the DTO shape the frontend already consumes via `use-plan.hooks.ts` is identical; only its runtime population changes. There is no file overlap and no ordering dependency in either direction — land this whenever convenient relative to Slices 2-7.

**What Slice 2's Today header can honestly show once this lands.** Slice 2's D6 (`slice-2-today.md`) builds `WEEK N OF M` off the existing `resolveCurrentWeek` and treats its output as "the current week, honestly" — that phrasing is accurate *after* this ships, not fully accurate before it. Today, `resolveCurrentWeek`'s clamp ceiling (`lastWeek`) never moves past 4, so any plan past week 4 shows a permanently stale header and week grid regardless of how this design's rolling horizon lands. Once the sweeper is running, `lastWeek` grows as real weeks populate, so in steady state Slice 2's header and THE WEEK grid render the runner's actual current week with real workouts — with zero code change required on Slice 2's side; it was already written to consume the clamp's output as ground truth.
